# 单服务器部署100个站点 - 三种方案对比

## 方案概述

基于您的 Next.js + MySQL 架构，以下是三种可行的部署方案，各有优劣。

---

## 方案一：单应用实例 + 多数据库（推荐）

### 架构设计
- **1个 Next.js 应用实例**（单进程）
- **100个独立 MySQL 数据库**（每个站点一个数据库）
- **Nginx/Caddy 反向代理**（根据域名路由）
- **动态数据库连接**（根据请求域名切换数据库）

### 实现原理
1. 通过中间件识别请求的域名（Host header）
2. 根据域名映射表动态切换 Prisma 数据库连接
3. 所有站点共享同一份代码和进程资源

### 优点
✅ **资源占用最少**：只需1个Node.js进程，内存占用低  
✅ **部署简单**：只需部署一次应用代码  
✅ **维护方便**：代码更新一次，所有站点生效  
✅ **扩展性好**：新增站点只需创建新数据库和域名配置  
✅ **成本最低**：服务器资源利用率高  

### 缺点
⚠️ **单点故障风险**：应用崩溃影响所有站点  
⚠️ **数据库连接数限制**：需要合理配置连接池  
⚠️ **代码隔离性差**：所有站点必须使用相同代码版本  

### 技术实现要点
- 使用 Prisma 的动态数据源切换
- 实现域名到数据库的映射中间件
- 配置 Nginx 反向代理规则
- 使用连接池管理数据库连接

### 预估资源需求
- CPU: 2-4核
- 内存: 4-8GB
- 磁盘: 根据数据量（建议SSD）
- 数据库连接: 需要合理配置连接池大小

---

## 方案二：多进程 + 端口隔离 + 反向代理

### 架构设计
- **100个 Next.js 应用实例**（每个站点独立进程，不同端口）
- **100个独立 MySQL 数据库**（每个站点一个数据库）
- **Nginx/Caddy 反向代理**（根据域名路由到不同端口）
- **PM2/Systemd 进程管理**（管理100个进程）

### 实现原理
1. 每个站点运行在独立端口（如 3001-3100）
2. Nginx 根据域名将请求转发到对应端口
3. 每个进程独立运行，互不影响

### 优点
✅ **完全隔离**：每个站点独立进程，互不影响  
✅ **灵活性强**：可以为不同站点配置不同环境变量  
✅ **故障隔离**：单个站点崩溃不影响其他站点  
✅ **便于调试**：可以单独重启某个站点  

### 缺点
⚠️ **资源占用大**：需要100个Node.js进程，内存占用高（每个约50-100MB）  
⚠️ **部署复杂**：需要管理100个进程的启动、停止、更新  
⚠️ **维护成本高**：代码更新需要重启100个进程  
⚠️ **端口管理**：需要管理100个端口号  

### 技术实现要点
- 使用 PM2 集群模式或独立进程管理
- 编写自动化脚本管理多进程
- 配置 Nginx upstream 和 server 块
- 使用环境变量区分不同站点配置

### 预估资源需求
- CPU: 8-16核（推荐）
- 内存: 16-32GB（每个进程约50-100MB）
- 磁盘: 根据数据量
- 进程管理: PM2 或 Systemd

---

## 方案三：Docker 容器化部署（推荐用于生产环境）

### 架构设计
- **100个 Docker 容器**（每个站点一个容器）
- **100个独立 MySQL 数据库**（可在容器内或外部数据库服务器）
- **Nginx/Caddy 反向代理**（根据域名路由到不同容器）
- **Docker Compose 编排**（统一管理所有容器）

### 实现原理
1. 每个站点打包成独立的 Docker 镜像
2. 使用 Docker Compose 管理100个容器
3. Nginx 根据域名路由到对应容器
4. 数据库可以容器化或使用外部数据库服务器

### 优点
✅ **环境隔离**：每个站点完全独立的运行环境  
✅ **易于扩展**：可以轻松迁移到多服务器  
✅ **版本管理**：可以为不同站点使用不同代码版本  
✅ **资源限制**：可以为每个容器设置CPU/内存限制  
✅ **标准化部署**：Docker 镜像便于版本控制和回滚  
✅ **便于监控**：可以使用 Docker 监控工具  

### 缺点
⚠️ **资源占用较大**：Docker 容器有一定开销  
⚠️ **学习曲线**：需要掌握 Docker 和 Docker Compose  
⚠️ **存储占用**：需要存储100个镜像（可通过多阶段构建优化）  
⚠️ **网络配置**：需要配置 Docker 网络和端口映射  

### 技术实现要点
- 编写 Dockerfile 优化镜像大小
- 使用 Docker Compose 管理多容器
- 配置 Nginx 反向代理到 Docker 容器
- 使用 Docker volumes 管理数据持久化
- 考虑使用 Docker Swarm 或 Kubernetes（如果未来需要）

### 预估资源需求
- CPU: 8-16核
- 内存: 16-32GB
- 磁盘: 根据数据量 + 镜像存储（建议100GB+）
- Docker: 需要足够的存储空间

## 推荐选择

### 🥇 **首选：方案一（单应用+多数据库）**
- **适合场景**：站点规模相似、资源有限、需要快速部署
- **理由**：资源占用最少，维护成本最低，对于100个相似站点最经济

### 🥈 **次选：方案三（Docker容器化）**
- **适合场景**：生产环境、需要标准化、未来可能扩展
- **理由**：平衡了资源占用和隔离性，便于未来迁移和扩展

### 🥉 **备选：方案二（多进程）**
- **适合场景**：需要完全隔离但不想使用容器
- **理由**：资源占用较大，但隔离性好，适合对隔离要求高的场景
